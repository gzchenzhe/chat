## 1. 给下一位 AI 的快速结论

这是一个**无需构建工具的纯静态 Vue 3 PWA**，用于编辑并导出仿微信聊天界面的 PNG 图片。项目的 HTML、CSS 和业务 JavaScript 基本都集中在 `index.html`；依赖、字体和界面图片均已放在本地，正常运行时不依赖 CDN。

接手时按以下顺序阅读：

1. 本文档：了解真实进度、风险和接手规则。
2. `README.md`：了解产品功能和安装方式。
3. `index.html`：唯一的主要界面与业务入口。
4. `sw.js` 和 `manifest.webmanifest`：PWA 缓存与安装配置。

当前没有 `.git` 目录，无法从提交记录可靠还原改动原因、作者或历史进度。本文描述的是 2026-07-15 实际文件状态，不把文件时间或旧日志当作开发历史。

## 2. 项目目标与当前完成度

### 已实现

- 三页面式移动端界面：`首页 / 编辑器 / 预览分享`，由 Vue 状态 `activePage` 切换，不使用路由。
- 浅色、深色两种聊天主题。
- 状态栏编辑：时间、返回 App 文案、标题、未读数、电量、Wi-Fi/5G/4G/3G 网络标识。
- 人物设置：双方头像、对方昵称及昵称显示开关。
- 八类聊天内容：文字、图片、语音、转账、通话、拍一拍、时间戳、红包。
- 消息的添加、编辑、删除与 HTML5 拖拽排序。
- 固定尺寸聊天预览：内容画布为 `375 × 812`，外层带手机边框。
- 使用 `localStorage` 自动保存编辑数据。
- 两阶段图片导出：先生成最终图片并显示缩略图，确认后再下载 PNG；数据变化后会把旧生成图标为过期并阻止直接导出。
- 导出时等待字体和图片，使用 3 倍像素比，并单独重绘头像和聊天图片，以规避部分 `html-to-image` 图片渲染问题。
- PWA Manifest、Service Worker 离线缓存、主屏幕图标和 IIS 配置均已存在。
- Vue、Tailwind CSS、`html-to-image`、字体和界面素材均为本地文件。

### 当前判断

产品主要链路已经写完，可视为“功能型 MVP”。但目前只能确认代码和资源在静态层面完整，不能把它描述为已经完成生产验收：仓库中没有测试脚本、测试报告、浏览器兼容性矩阵或部署环境记录。

## 3. 技术结构

| 项目 | 当前实现 |
| --- | --- |
| 应用形态 | 纯静态单页 PWA |
| 框架 | Vue `3.5.35`，通过本地生产版全局脚本加载 |
| 样式 | `index.html` 内联 CSS + 本地编译后的 Tailwind CSS |
| 截图 | 本地 `html-to-image`，输出 PNG |
| 数据 | 浏览器 `localStorage`，无后端、无数据库 |
| 构建 | 无 `package.json`、无打包器、无构建步骤 |
| 部署 | 任意静态 HTTP/HTTPS 服务；另有最小化 IIS `web.config` |
| PWA | `manifest.webmanifest` + `sw.js` |

核心数据流：

```text
用户编辑
  -> Vue 响应式状态
  -> watcher 自动写入 localStorage
  -> 聊天预览即时更新
  -> 生成图片（html-to-image + 图片覆盖层重绘）
  -> 内存中的 PNG Data URL
  -> 用户确认缩略图后下载
```

- 未读数 `92`、电量 `65`、网络 `5G`。
- 一条对方发送的“欢迎使用”文字消息。

### 本地存储

- 键名：`wechat_editor_state_v18`。
- 保存内容：主题、状态栏字段、双方头像、昵称设置和全部消息。
- 不保存：当前页面、工具展开状态、已经生成的 PNG 和导出面板状态。
- 头像和聊天图片以 Data URL 保存，较大的图片可能触发浏览器约 5 MB 的存储上限；代码已捕获 `QuotaExceededError` 并提示用户。
- 当前没有数据结构迁移逻辑。若改变消息字段或存储键，需要决定是否兼容旧数据，不要直接覆盖版本号后假设历史数据可用。

### PWA 缓存

- 当前缓存名：`wechat-screenshot-pwa-v29`。
- 安装时预缓存入口、Manifest、图标、主要依赖、界面图片和实际使用的字体。
- 导航请求采用网络优先，失败时回退到缓存的 `index.html`；其他 GET 请求采用缓存优先。
- 更新页面或资源后，如果安装版仍显示旧内容，先提升 `CACHE_NAME`，再测试普通刷新、已安装 PWA 和离线启动。

## 6. 图片导出链路

当前绑定在界面上的主流程是：

1. `generatePreparedImage()` 切换到预览页并最多尝试生成三次。
2. `createExportImageDataUrl()` 等待资源，收集图片覆盖层，暂时隐藏原图片，再以 `pixelRatio: 3` 生成基础图。
3. `paintImageOverlays()` 在 Canvas 中按裁剪范围、圆角、透明度和 `object-fit` 重绘头像/聊天图片。
4. 结果写入内存状态 `generatedImageUrl`，缩略图显示后才允许 `exportGeneratedImage()` 下载。
5. 任一可保存字段变化时，`generatedImageDirty` 变为 `true`，旧图必须重新生成。

`index.html` 末尾还保留了旧的 `generateImage()` 及 `shareOrDownloadImage()` 流程，其中包含 Web Share API 回退逻辑，但当前模板没有调用 `generateImage()`。不要在没有回归测试的情况下直接删除；也不要把“系统分享”写成当前主流程已经确认可用的功能。README 中“预览分享”的说法与当前主要按钮的下载行为需要在产品需求明确后统一。

## 7. 本地运行

不要直接双击 `index.html` 使用 `file://` 打开，因为 Service Worker、资源加载和部分截图行为需要 HTTP 环境。

在项目根目录执行：

```bash
python -m http.server 8188
```

然后访问：

```text
http://127.0.0.1:8188/
```

PWA 正式部署应使用 HTTPS；`localhost`/`127.0.0.1` 仅适合本地开发。

## 9. 已知风险与技术债

1. **单文件过大**：业务、模板和样式集中在约 2392 行的 `index.html`，改动容易产生连锁影响。
2. **缺少版本控制信息**：当前目录不是 Git 仓库，无法安全查看差异、回滚或追踪决策。
3. **缺少自动化质量门禁**：没有 lint、单元测试、端到端测试或 CI。
4. **移动端拖拽待验证**：排序使用 HTML5 `dragstart/drop` 事件，触屏 Safari/Chrome 的可用性不能仅凭桌面行为推断。
5. **导出兼容性风险**：字体、Canvas、图片解码、跨域和浏览器内存都可能影响最终 PNG；复杂的图片覆盖层逻辑说明这里是高风险修改区。
6. **本地存储容量有限**：多个大图会迅速占满 `localStorage`，且清理站点数据后无法恢复。
7. **无数据迁移**：存储键带版本号，但没有旧数据升级策略。
8. **生成图不持久化**：刷新页面后需要重新生成最终 PNG，这是当前设计，不应误判为保存失败。
9. **旧导出代码仍保留**：主流程与遗留流程并存，后续重构前需先建立回归基线。
10. **资源授权待确认**：第三方脚本、字体和仿微信素材没有在当前目录中看到完整的许可证/来源清单，公开发布前应审查授权和品牌使用边界。
